▌ Q1: ขอบเขตการรื้อใหม่ — แค่ UI หรือรวม backend ด้วย?
  สถานะ: ✓ ตอบแล้ว 19 มิ.ย. 2569 → "Step-by-step (A→C ใน UI-only ก่อน)"
  ผมเห็น 3 ทางเลือก:
    (A) UI-only — ทำหน้าตามดีไซน์ แต่ใช้ data จาก /api/auth/me เดิม
        + แก้ profile ทีเดียวก็พอ ไม่มี table ใหม่
        → เร็ว 1-2 วันเสร็จ แต่ feature ในดีไซน์ใช้ไม่ครบ
    (B) UI + Tables ใหม่ — เพิ่ม insurance_plans, articles, contacts
        → ครบตามที่ลูกค้าวาดในเอกสาร แต่ใช้เวลา 5-7 วัน
        → ต้อง migrate DB บน HostAtom
    (C) Step-by-step — ทำ (A) ก่อน deploy ให้ลูกค้าดู, ค่อยเพิ่ม (B) ทีหลัง

  ★ ผมแนะนำ (C) — ลูกค้าได้เห็นเร็ว, ไม่เสี่ยงพังของเดิม

▌ Q2: Public landing — แสดงข้อมูลใคร?
  สถานะ: ✓ ตอบแล้ว 19 มิ.ย. 2569 → "กวิสรา คนเดียว เป็น showcase (A)"
  ดีไซน์ S__13730279.jpg เป็นหน้าโปรไฟล์ของ "กวิสรา จงเจริญ" คนเดียว
  แต่ระบบปัจจุบันรองรับหลายตัวแทน (search.html ค้นหาได้)

  ตัวเลือก:
    (A) Public หน้าแรก = profile ของกวิสรา (single agent showcase)
        + เพิ่มลิงก์ "ดูตัวแทนคนอื่น" ไป search
    (B) Public หน้าแรก = directory ของทีม VP Kanniporn
        แต่ละตัวแทนมีหน้าของตัวเองที่ /agent/:id
    (C) ตัวแทนใหญ่ (กวิสรา) เป็น home; ทีมงานเป็น /agent/:id

  ★ ผมแนะนำ (A) เพราะดีไซน์ตรงนี้ + เป็นเว็บส่วนตัวของกวิสรา
    ส่วนทีมงานเข้าผ่าน /search

▌ Q3: ลำดับงาน — ทำหน้าไหนก่อน?
  สถานะ: ✓ ตอบแล้ว 19 มิ.ย. 2569 → "Public landing → Profile → Admin (A)"
  สิ่งที่ทำได้:
    [a] Public landing (หน้าแรกที่ลูกค้าเห็น) ← แนะนำเริ่มที่นี่
    [b] หน้าโปรไฟล์ /agent/:id (ตาม PDF หน้า 2)
    [c] Admin dashboard layout (ตาม S__13730278.jpg)
    [d] Plans management (ตาม PDF หน้า 4-6)
    [e] Edit profile ใหม่ (ตาม PDF หน้า 7)

  ★ ผมแนะนำ a → b → c → e → d (เห็นผลเร็ว, รอ DB migration ทีหลัง)

▌ Q4 (เสริม ไม่ blocking): หมวดหมู่แบบประกัน — มีกี่หมวด?

    2. ประกันสะสมทรัพย์ (Savings) — มี: ระยะสั้น, ระยะยาว
    3. ประกันบำนาญ (Retirement) — มี: ระยะสั้น, ระยะยาว
