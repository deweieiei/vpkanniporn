  1. ⚠️ Production รัน **MariaDB** ไม่ใช่ MySQL !!!
     - docker-compose.yml เขียน `mysql:8.0` → **ไฟล์นั้นโกหก อย่าเชื่อ**
     - AI_Memory เก่าจดว่า MySQL → **ผิด**
     - รู้ได้จาก error ของ phpMyAdmin ที่ขึ้นว่า "your MariaDB server version"
     - ผลกระทบจริงที่เจอ: `CAST(? AS JSON)` **ใช้ไม่ได้บน MariaDB** (ERROR 1064)
       → เขียน JSON ลง DB ให้ส่งเป็น string ธรรมดา อย่าใช้ CAST
       (MySQL แปลงให้เอง / MariaDB มองว่า JSON = LONGTEXT)
     - ที่ใช้ได้ปกติ: JSON_OBJECT, JSON_ARRAY, JSON_EXTRACT, JSON_UNQUOTE,
       ON DUPLICATE KEY UPDATE, FROM DUAL WHERE NOT EXISTS, self-FK CASCADE

  2. ⚠️ DB user บน HostAtom **อ่าน information_schema ไม่ได้**
     - `#1044 - Access denied ... to database 'information_schema'`
     - → ห้ามใส่ SELECT จาก information_schema ในไฟล์ SQL ที่ส่งให้พี่ดิวรัน
     - อยากเช็คโครงสร้าง ใช้ `SHOW COLUMNS` / `SHOW TABLES` แทน

  3. ⚠️ CREATE TABLE IF NOT EXISTS = กับดัก
     - ถ้าเคยรันไฟล์เวอร์ชันเก่าไปแล้ว ตารางจะถูกสร้างด้วยกฎเก่าและ
       "IF NOT EXISTS" จะข้ามการแก้ให้ → เจอ ERROR 1048 cannot be null
     - แก้ schema ที่รันไปแล้ว ต้อง ALTER หรือ DROP+CREATE เท่านั้น

  4. ⚠️ เวลาส่ง SQL ให้พี่ดิว — ทำให้ "ก็อปวางทั้งไฟล์แล้วจบ"
     - พี่ดิวเคยก็อปมาแค่บางท่อน → มี `)),;` ห้อยท้าย → syntax error
     - → แยก INSERT เป็นคำสั่งละแถว อย่ารวมเป็นก้อน VALUES ยาวๆ
     - → ไฟล์ยิ่งสั้นยิ่งดี ตัดคอมเมนต์ที่ไม่จำเป็นออก
     - `db/insert_blocks_only.sql` (46 บรรทัด ไม่มีคอมเมนต์เลย) คือตัวที่สำเร็จ

  5. ⚠️ เทสต์ไทย/อีโมจิผ่าน curl ใน Git Bash บน Windows = ได้ ????? เสมอ
     - **นั่นคือ shell ทำเพี้ยน ไม่ใช่แอปพัง** — อย่าไป "แก้" pool.js
     - เทสต์ให้ถูก: เขียน JSON ลงไฟล์ แล้วใช้ `curl --data-binary @file.json`

  แนวคิด: หน้าตาเว็บ (หน้าแรก) = รายการ "บลอค" เรียงกัน
          SupperAdmin คลิกแก้บนหน้าจริงได้เลย (contenteditable + เซฟตอนคลิกออก)
          เพิ่มฟีเจอร์ใหม่ = ใส่ key ใน JSON ไม่ต้อง ALTER TABLE อีกต่อไป

  ★ ทำไมถึงต้องรื้อ: ตาราง users เคยเป็นถังขยะ — มี ~35 คอลัมน์ และ alter 14 ไฟล์
    เพราะทุกฟีเจอร์ใหม่ = ADD COLUMN อีกที + ลำดับ section ตายตัวใน HTML

  ┌─ ตาราง page_blocks ────────────────────────────────────┐
  │ user_id    NULL = บลอคของหน้าตาเว็บ (หน้าแรก) ← ใช้ตอนนี้│
  │            = N  = บลอคของหน้าโปรไฟล์ตัวแทนคนที่ N       │
  │                   (เผื่ออนาคต ยังไม่ได้ใช้)              │
  │ parent_id  บลอคแม่ (NULL = ชั้นบนสุด) ซ้อนได้ไม่จำกัด    │
  │ type       ชนิดบลอค                                     │
  │ sort_order ลำดับบนหน้า (สลับลำดับ = แก้เลขนี้)          │
  │ is_visible แสดง/ซ่อน                                    │
  │ data       JSON — เนื้อหา (รูปแบบต่างกันตาม type)        │
  └────────────────────────────────────────────────────────┘

  ┌─ ตาราง block_images ("ภาพบลอค") ───────────────────────┐
  │ slot     ภาพที่เท่าไหร่ (0-5 = 6 ภาพ)                    │
  │ variant  web = คอม / tablet = iPad / mobile = โทรศัพท์  │
  │ → hero 1 บลอค = 6 slot × 3 variant = อัปได้ 18 รูป      │
  │   (พี่ดิวสั่งชัดเจน: "เอา18รุปเลย 6ภาพแยกเป็นส่วน 3 device")│
  │ ไม่มี variant ไหน → ระบบ fallback ไปใช้ 'web' อัตโนมัติ  │
  │ UNIQUE (block_id, slot, variant) → อัปทับช่องเดิมได้     │
  └────────────────────────────────────────────────────────┘

  ชนิดบลอคของหน้าแรก (ตรงกับที่พี่ดิวสั่งมา 8 ข้อ):
    1. hero     ภาพหัวเว็บ 6×3 + ข้อความ + ปุ่ม
    2. plans    "เลือกแผนที่เหมาะกับคุณ" → ลูกเป็น type='page'
       page     1 ปุ่ม = 1 หน้า กดเข้าไปสร้างบลอคย่อยได้อีก ไม่จำกัดชั้น
                ชั้นในสุดเขียนโพสยาวๆ ได้ (data.content = HTML ผ่าน sanitize)
    3. features "ทำไมต้องวางแผนกับเรา" → ลูกเป็น type='card'
       card     icon + ตัวหนังสือใหญ่ (เลือกสีได้) + คำอธิบายยาวๆ
    4. agents   "รู้จักที่ปรึกษาของคุณ" โชว์ตัวแทน 3-5 คนเป็น popcard
                แบบหน้าค้นหา — แอดมินเลือกเองว่าจะโชว์ใคร (data.agent_ids)
    5. awards   "ความสำเร็จและรางวัล" → ลูกเป็น type='card' (โครงเดียวกับข้อ 3)
    6. (ไม่มี reviews — พี่ดิวสั่งเอาออก 16 ก.ค.)
    7. recruit  "กำลังมองหาอาชีพ..." ใส่ภาพอย่างเดียว
    8. cta      "พร้อมเริ่มต้นวางแผนกับเราไหม?" LINE + เบอร์โทร + นัดหมาย
                data.appointment_agent_id = ใบนัดหมายเข้าให้ตัวแทนคนไหน
                → ลง contact_inquiries kind='appointment' → ตัวแทนดูที่ /inquiries
    เสริม: text = ข้อความอิสระ / image = รูปอิสระ (มี type รองรับแล้ว ยังไม่มี UI)

  - พี่ดิว commit + push GitHub เอง (ทุก commit ชื่อ "up") — เราไม่ต้อง commit ให้
    ★ พี่ดิวอาจ push ระหว่างที่เรากำลังแก้อยู่ → เช็ค git log ก่อนสรุปอะไร
  - พี่ดิวไม่ชอบงานใหญ่โต ("ฉันพยายามไม่ให้นายทำใหญ่โต เดี๋ยวจะเหนื่อย")
    → ทำเท่าที่จำเป็น อย่าขยายขอบเขตเอง

  ★★ บทเรียนใหญ่สุดของรอบนี้ ★★
  ตอนพี่ดิวบอกว่า "ออกทะเลละมั่วไปหมด" — ผมเข้าใจผิดว่าดีไซน์ผิด แล้วเสนอรื้อทิ้ง
  ความจริงพี่หมายถึง "SQL มันรันแล้วผิดซ้ำๆ เลยมองภาพไม่ออก"
  พอ SQL ผ่าน พี่เห็นของจริงปุ๊บ → "มันสุดยอด"
  → บทเรียน: ถ้าพี่ดิวหงุดหงิด ให้ถามว่าหงุดหงิดเรื่องอะไรก่อน อย่าเดาว่าเป็นเรื่องดีไซน์
  → และ: ให้พี่ "เห็นของจริง" เร็วที่สุด ดีกว่าอธิบายยาวๆ ว่าเราทำอะไร
