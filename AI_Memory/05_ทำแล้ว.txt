  ★ พี่ดิวรายงาน: หน้า /agent/2 ไม่อัปเดตตามที่แก้ใน /dashboard
    console error จริงจาก production:
      PUT https://www.vpkanniporn.com/api/auth/profile → 500
      {error: "เกิดข้อผิดพลาด กรุณาลองใหม่"} (ข้อความจริงถูกซ่อนเพราะ NODE_ENV=production)

  ◉ หาสาเหตุจาก client-side (public/dashboard.html) เพราะข้อความ error จริง
    ถูกซ่อนไว้ (ต้องรอ log จาก Plesk ถ้าจะดูของจริง แต่เจอ bug ชัดเจนจากโค้ดก่อน)

  ★ Root cause เจอแล้ว — 100% ชัวร์ ไม่ต้องเดา:
    ฟังก์ชัน saveProfile() (dashboard.html บรรทัด ~937) เดิม:
      fd.append('first_name', currentUser.first_name || '');   // (1) เขียนก่อนเสมอ
      fd.append('last_name', currentUser.last_name || '');
      for (const [k, v] of Object.entries(updates)) fd.append(k, v); // (2) เขียนทับ

    ปุ่ม "บันทึกโปรไฟล์" (form submit บรรทัด ~989) ส่ง updates ที่มี
    first_name/last_name อยู่แล้ว → fd.append ถูกเรียกซ้ำ 2 ครั้งสำหรับ
    key เดียวกัน → FormData มี 2 ค่าใน field เดียวกัน
    → multer parse เป็น req.body.first_name = ["", "ชื่อใหม่"] (array แทน string)
    → pool.query('UPDATE users SET first_name = ?, ...', [arrayค่านี้, ...])
      mysql2 แปลง array parameter เป็น comma list (สำหรับ IN clause)
      แต่ context นี้เป็น SET column = ค่าเดียว → กลายเป็น SQL ผิด syntax
      → throw error → next(err) → 500 (โดน NODE_ENV=production ซ่อนข้อความ)

    ★ เกิดกับทุกครั้งที่กด "บันทึกโปรไฟล์" (มี first_name/last_name ใน updates)
      ไม่เกิดกับ "บันทึกผลงาน" (awards) เพราะ updates ไม่มี first_name/last_name ซ้ำ

  ◉ แก้แล้ว (dashboard.html):
  ✓ เปลี่ยนเป็น merge เป็น plain object ก่อน (`{...default, ...updates}`)
     แล้วค่อย fd.append ทีเดียวต่อ 1 key — ห้ามซ้ำ
  ✓ ทดสอบ logic การ merge ผ่าน preview_eval จำลอง FormData.getAll()
     ยืนยันว่าแต่ละ field เหลือค่าเดียวแล้ว (ไม่ใช่ array)
  ✗ ยังทดสอบ end-to-end ผ่าน login+save จริงไม่ได้ (ไม่มี MySQL local)

▌ 2026-07-03 (รอบ 6 — session store persistent) — Claude Sonnet 5
  ★ พี่ดิว: "ต้องการให้จำการ login"

  ◉ วินิจฉัยสาเหตุ:
    ก่อนหน้านี้ session cookie ตั้ง maxAge 30 วันไว้แล้ว (ทำตั้งแต่รอบก่อนๆ)
    แต่ session middleware ไม่ได้ตั้งค่า `store` → ใช้ default MemoryStore
    ของ express-session ซึ่งเก็บ session ไว้ใน RAM ของ process เดียว
    → ทุกครั้งที่ Node process restart (deploy ใหม่, crash, หรือแม้แต่
      `node --watch` ตอน dev ที่ restart เองเวลาไฟล์เปลี่ยน) session หายหมด
      ผู้ใช้ต้อง login ใหม่ทั้งที่ cookie ยังไม่หมดอายุ ← นี่คือสาเหตุที่ "ไม่จำ login"

  ◉ แก้ไข (index.js):
  ✓ เพิ่ม dependency `express-mysql-session` (^3.0.3) — session store ที่เก็บ
     ลง MySQL table `sessions` แทน RAM ใช้ pool เดิมจาก db/pool.js (ไม่เปิด
     connection ซ้ำ) → session อยู่รอดข้าม process restart ได้จริง
  ✓ ตั้ง expiration: 30 วัน ตรงกับ cookie.maxAge เดิม
  ✓ createDatabaseTable: true → table `sessions` จะถูกสร้างอัตโนมัติตอน
     boot ครั้งแรก (ไม่ต้องรัน SQL เพิ่มเอง ไม่ต้องแก้ schema.sql)
  ✓ เพิ่ม `.on('error', ...)` กัน process crash ถ้า DB ต่อไม่ได้ตอน boot
     (EventEmitter 'error' ที่ไม่มี listener จะทำให้ Node throw ทั้ง process)
  ✓ graceful shutdown เพิ่ม `sessionStore.close()` ก่อน `pool.end()`
     (ปิด timer เคลียร์ session หมดอายุ โดยไม่ปิด pool เพราะใช้ร่วมกับ query อื่น
     — เช็ค source code ของ lib แล้วว่า endConnectionOnClose=false เมื่อ
     ส่ง pool เข้าไปเอง ปลอดภัย ไม่ปิด pool ซ้ำ)

  ◉ ทดสอบ (เครื่องนี้ยังไม่มี Docker/MySQL local — ทดสอบเท่าที่ทำได้):
  ✓ npm run dev บูตสำเร็จ ไม่ crash แม้ DB ต่อไม่ได้ (error listener ทำงาน)
  ✓ GET / และ GET /api/auth/me ตอบ 200 ปกติ (ไม่ต้องอ่าน session store
     ถ้ายังไม่มี cookie เดิม)
  ✓ POST /api/auth/login ตอบ 500 เร็ว (18ms) ไม่ hang — fail fast ตามคาด
     เพราะ query DB ไม่ได้ (พฤติกรรมเดิมก่อนแก้ก็เป็นแบบนี้ ไม่ได้แย่ลง)
  ✗ ยังไม่ได้ทดสอบ "login แล้ว restart server แล้วยัง login ค้างอยู่มั้ย"
     จริงๆ เพราะต้องมี MySQL ก่อน (รอ Docker Desktop)

  ▌ ทำต่อรอบหน้า (หลังพี่ดิวลง Docker + เปิด docker compose up -d):
    1. Register/login ทดสอบให้เห็น table `sessions` ถูกสร้างอัตโนมัติใน DB
    2. Restart dev server (kill + npm run dev ใหม่) แล้วเช็คว่า refresh
       browser แล้วยัง login อยู่ (ไม่ redirect ไป /login) — คือการยืนยันจริง
       ว่า "จำ login" ทำงาน
    3. Deploy ขึ้น HostAtom แล้วเช็คว่า table sessions ถูกสร้างบน production DB
       ด้วย (createDatabaseTable: true จะสร้างให้อัตโนมัติเหมือนกัน)
